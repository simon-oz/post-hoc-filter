1) What signals does ensemble_decision combine?
For each (query, result_text) pair (where result_text = title + " " + snippet), the function builds four signals:


Keyword prefilter (boolean)

Quick rule: if none of the meaningful query terms (≥4 chars) appear in the title+snippet, the item is obviously Irrelevant → return "I" immediately.
This avoids wasting compute on “How to Bake a Cake” for an ML query.



Embedding similarity (float)

Compute cosine similarity between query and title+snippet using your embedding model (e.g., gemma‑300m).
Map that score to a label with thresholds:

score ≥ embed_R → R
embed_P ≤ score < embed_R → P
score < embed_P → I





Zero‑shot NLI (label + score)

Use an NLI model (e.g., MoritzLaurer/deberta-v3-large-zeroshot-v2.0) with the hypothesis template:
“This example is {label} to the query: {query}.”
Returns a top label among ["Relevant","Partially Relevant","Irrelevant"] and a confidence score.
We then apply acceptance thresholds:

If top is R and score ≥ nli_R → accept R
If top is P and score ≥ nli_P → accept P
If top is I → accept I (we treat I as “confident by default”)



If the top label doesn’t meet its threshold, we treat NLI as “no strong opinion” (nli_accepted = None).


(Optional) LLM fallback

When signals disagree or are borderline, a small, strict prompt is sent to a local LLM.
The model returns exactly one of R/P/I, and we use that result.




2) The decision rules (in plain English)
Let:

embed_lbl ∈ {R, P, I} from the embedding thresholds
nli_accepted ∈ {R, P, I, None} from NLI + thresholds
s_embed = the raw cosine similarity

Then:

Keyword prefilter fails → I
Strong agreement:

If embed_lbl = R and nli_accepted = R → R
If embed_lbl = I and nli_accepted = I → I


Direct conflict R vs I (one says R, the other says I) → P (conservative middle)
Both say P → P
NLI accepts I, but embeddings are very high (≥ embed_R + 0.10) → P (hedge) else I
NLI accepts R, but embeddings only P or I:

If s_embed ≥ embed_P and NLI is very confident (≥ nli_R + 0.10) → R
Else → P


If still ambiguous → use LLM fallback (strict single‑letter output)
If fallback disabled → default P (conservative)


This policy is conservative: we try to avoid false R by requiring agreement or strength.


3) Typical thresholds & how to read them
Default starting points (tune on a small dev set):
Plain Text embed_R = 0.55embed_P = 0.35nli_R   = 0.60nli_P   = 0.45Show more lines

Embedding thresholds are model‑dependent; for many sentence‑embedding models, 0.55/0.35 are reasonable starting splits for R/P/I.
NLI thresholds prevent over‑confident “R” (require ≥0.60) but allow moderate “P” (≥0.45).

5) What the diagnostics object means
Each result gets a small dict you can log to understand “why”:
JSON{  "prefilter": 1.0,         // 1.0 passed; 0.0 failed  "embed_score": 0.57,      // cosine similarity (normalized embeddings)  "nli_label": "P",         // raw top NLI label  "nli_score": 0.52         // NLI confidence for the top label}Show more lines

prefilter = 0.0 means rule 1 fired → output I without others.
embed_score shows where you stand relative to embed_R / embed_P.
nli_label/nli_score show what NLI thought, and whether it met the thresholds (nli_R / nli_P).

These values are extremely useful for threshold tuning.

6) How to tune thresholds

Prepare 20–100 labeled examples (your new “Reference answer” field is perfect).
Grid search small ranges:

embed_R: 0.52–0.62 by 0.02
embed_P: 0.30–0.40 by 0.02
nli_R:   0.58–0.68 by 0.02
nli_P:   0.40–0.50 by 0.02


For each setting, compute:

Overall accuracy
Per‑class agreement (R/P/I recalls)
Macro‑F1
Cohen’s κ


Pick the setting that maximizes what you care about (e.g., high R recall with acceptable I precision, or balanced Macro‑F1).


Practical tip: keep embed_P about 0.20 below embed_R, and keep nli_R − nli_P ≈ 0.15 so R requires meaningfully more confidence than P.


7) Edge cases & recommended policies

Very short queries (“React”) → prefilter may be too strict. Consider lowering the min token length (e.g., from 4 to 3).
Ambiguous snippets (marketing fluff) → embeddings may be optimistic; let NLI or fallback gate R.
Domain drift (e.g., biomedical) → raise nli_R a bit to avoid spurious R.
Performance: If NLI is heavy, you can disable fallback and rely on “Rule 6” with a higher nli_R to still get good R decisions.

  (query, text) → | Keyword prefilter?   | -- no --> I
                  +----------------------+
                            |

                            v

                  | Embedding similarity |

   - Both R → R         - Both I → I       - R vs I → P

   - Else → fallback (LLM) or P


9) TL;DR

Prefilter removes obvious junk (I).
Embeddings provide a smooth “semantic relatedness” score (R/P/I via thresholds).
NLI brings a logical “is this related?” judgment with confidence; it can promote R or demote to I.
Rules reconcile conflicts conservatively; fallback resolves tough ties.
Diagnostics + threshold tuning help tailor performance to your domain.

=========================

The local LLM is NOT called for every decision.
Even when use_llm_fallback = True (by default, unless simply add --use_llm_fallback in CLI to disable fallback entirely), the local LLM is only called when needed—i.e., when the other signals are ambiguous or conflict.
In your ensemble_decision(...), the LLM is skipped in these common cases:

Keyword prefilter fails → immediate I (no LLM).
Embeddings & NLI agree strongly → immediate R or I (no LLM).
Clear rules decide P (e.g., direct conflict R vs I) → (usually) no LLM.

The LLM fallback is invoked for tricky cases, such as:

NLI top label doesn’t meet its acceptance threshold (nli_accepted = None) and embeddings aren’t decisive.
NLI says I but embeddings are very high (we hedge to P or consult LLM).
NLI says R but embeddings are low unless NLI is very confident.
Any residual borderline scenario after the rules.

This design keeps the pipeline efficient and conservative: the LLM is a tie‑breaker, not the main classifier.
